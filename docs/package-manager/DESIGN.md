## 1. Executive Summary

The Salt package manager (`sp`) provides Git-based dependency management, deterministic build ordering, and cross-package Z3 contract verification for Salt projects. It is designed for v0.1.0 to require zero server infrastructure — registries are plain Git repositories with a well-known tag convention.

### Design Goals

| Goal | Why |
|------|-----|
| **No server infrastructure** | The registry protocol is Git-only. Publish by pushing a tag. |
| **Deterministic builds** | Lockfile pins every transitive dependency to a precise commit. |
| **Cross-package verification** | A dependency's Z3 `requires` clauses are checked against each call site at compile time. |
| **Content-addressed cache** | Build artifacts are keyed by the hash of their inputs, not by version number. |
| **Gradual adoption** | A single-file Salt project with no `salt.toml` still compiles. `sp init` adds the manifest. |

### Non-Goals for v0.1.0

- Package publishing to a central registry (a la crates.io / npmjs.com)
- Private registry authentication
- WASM or cross-compilation targets via the package manager (the compiler handles `--target` independently)
- Dynamic linking of dependency packages

## 2. Package Manifest Format

Every Salt package has a `salt.toml` in its root directory. The file uses TOML v1.0 and is mandatory for any package that declares dependencies (a standalone single-file .salt with no imports does not need one).

### 2.1 Minimal Package

```toml
[package]
name = "my-app"
version = "0.1.0"
edition = "2026"
```

### 2.2 Full Manifest

```toml
[package]
name = "lettuce"
version = "0.3.0"
edition = "2026"
description = "A Redis-compatible in-memory cache written in Salt"
authors = ["Kevin McArthur"]
license = "MIT"
repository = "https://github.com/example/lettuce"

[dependencies]
# Git-based dependency — tag or branch
netd = { git = "https://github.com/example/netd", tag = "v0.2.0" }
# Branch reference (resolved to commit in lockfile)
keuos-std = { git = "https://github.com/example/keuos-std", branch = "main" }
# Rev pin for emergency override
zlib-salt = { git = "https://github.com/example/zlib-salt", rev = "a1b2c3d" }
# Path dependency (local development)
my-lib = { path = "../my-lib" }

[dev-dependencies]
test-harness = { git = "https://github.com/example/test-harness", tag = "v0.1.0" }

[build-config]
# Custom build flags passed to salt-front during `sp build`
saltc-args = ["--verify", "--target", "macos"]
```

### 2.3 Fields Reference

| Field | Section | Type | Required | Description |
|-------|---------|------|----------|-------------|
| `name` | `[package]` | string | yes | Package name, kebab-case, ASCII only |
| `version` | `[package]` | semver | yes | Semantic version (MAJOR.MINOR.PATCH) |
| `edition` | `[package]` | string | yes | Salt edition year; currently `"2026"` |
| `description` | `[package]` | string | no | One-line summary for tooling |
| `authors` | `[package]` | string[] | no | Author list |
| `license` | `[package]` | string | no | SPDX identifier |
| `repository` | `[package]` | string | no | Project homepage URL |

Dependency sources:

| Source | Fields | Example | Resolution |
|--------|--------|---------|------------|
| Git | `git`, plus one of `tag`, `branch`, `rev` | `{ git = "...", tag = "v0.2.0" }` | `git fetch` then resolve to commit SHA |
| Path | `path` | `{ path = "../my-lib" }` | Filesystem path; not cached |
| Dev | `[dev-dependencies]` | Same format | Only resolved during `sp test` |

### 2.4 Lockfile: `salt.lock`

Auto-generated by `sp build` and `sp add`. Must be committed to version control.

```toml
# salt.lock — DO NOT EDIT
version = 1

[[packages]]
name = "netd"
version = "0.2.0"
source = "git"
url = "https://github.com/example/netd"
commit = "e7f3a9b1c2d5e8f4a0b3c6d7e9f1a2b3c4d5e6f7"

[[packages]]
name = "keuos-std"
version = "0.5.0"
source = "git"
url = "https://github.com/example/keuos-std"
commit = "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0"
```

The lockfile stores the **resolved commit SHA** for every Git dependency so that `sp build` on another machine reproduces the exact same dependency graph. Path dependencies are NOT stored in the lockfile — they resolve at build time from the filesystem.

## 3. CLI Reference

The `sp` binary is a standalone CLI tool (implemented in Rust, separate from `salt-front`).

### 3.1 `sp new <path>`

Scaffolds a new package directory with `salt.toml` and a `src/main.salt` or `src/lib.salt`.

```bash
sp new my-project
cd my-project
# Creates: salt.toml, src/main.salt
```

### 3.2 `sp add <git-url>`

Fetches a Git dependency, resolves its version/tag, writes it into `salt.toml`, and runs `sp build` to verify the integration locally.

```bash
sp add https://github.com/example/netd
# Adds to [dependencies] in salt.toml
# Runs: git fetch, git checkout <tag>, read netd's salt.toml
# Runs: sp build --verify
# On failure: rolls back salt.toml and prints contract violations
```

**Contract pre-flight**: After fetching, `sp add` performs a trial build with Z3 verification on the combined dependency graph. If the dependency's public contracts contain `requires` clauses that cannot be satisfied at call sites within the current project, `sp add` **rejects** the dependency with an explanation.

### 3.3 `sp build`

Resolves the dependency graph, verifies all contracts, and compiles every package in topological order. Output goes to `salt-target/` (see Section 6).

```bash
sp build                   # Debug build
sp build --release         # Optimized build
sp build --target keuos    # Cross-compile for KeuOS
sp build --dry-run         # Print build order without compiling
```

Exit code 0 = success. Non-zero = resolution failure, verification failure, or compilation error.

### 3.4 `sp test`

Resolves dev-dependencies, then runs `sp build` followed by executing the test suite:

```bash
sp test                    # Run all tests
sp test -- --nocapture     # Pass through args to test runner
```

Test discovery: `sp test` finds all files matching `tests/*.salt` and `src/*_test.salt`, compiles each as a separate test binary, and runs them. A test is a Salt function annotated with `#[test]`:

```salt
#[test]
fn test_parse_empty_input() {
    // ...
}
```

### 3.5 `sp lock`

Regenerates `salt.lock` from scratch (re-resolves all version constraints). Useful when constraints change in `salt.toml`.

```bash
sp lock --check    # Fail if lockfile is stale (CI use)
```

### 3.6 `sp clean`

Removes the `salt-target/` directory.

```bash
sp clean
```

## 4. Dependency Resolution

`sp` uses the **PubGrub** algorithm (the same solver powering Dart's pub tool and, in adapted form, Cargo's resolver). PubGrub is chosen over the simpler SAT or backtracking approaches because it produces **high-quality error messages** — when resolution fails, the solver explains exactly which constraints conflict and why.

### 4.1 Why PubGrub

| Algorithm | Error Quality | Performance | Complexity |
|-----------|--------------|-------------|------------|
| Greedy (npm pre-v5) | None on conflicts | O(n) | Trivial |
| Backtracking (npm v5+) | Limited | O(2^n) worst case | Moderate |
| PubGrub | **Excellent**: pinpoints conflicting terms | O(n log n) typical | Moderate |
| SAT solver | Excellent | NP-complete in general | High |

PubGrub's key insight: instead of assigning versions blindly and backtracking, it maintains a **partial solution** (a set of decisions and their justifications) and detects incompatibilities incrementally. When a conflict is found, the algorithm derives a new **incompatibility** that summarizes the root cause and backjumps past irrelevant decisions.

### 4.2 Version Constraints

Salt uses semantic versioning (`MAJOR.MINOR.PATCH`). The constraint syntax in `salt.toml` follows the Cargo convention:

| Constraint | Meaning |
|------------|---------|
| `"0.2.0"` | Exactly 0.2.0 |
| `"^0.2.0"` | Compatible: `>=0.2.0, <0.3.0` |
| `"~0.2.0"` | Patch-compatible: `>=0.2.0, <0.2.1` |
| `">=1.0.0"` | At least 1.0.0 |
| `">=1.0.0, <2.0.0"` | Range intersection |

When no explicit version constraint is given alongside a Git URL, `sp add` reads the dependency's own `salt.toml` and uses its declared `version` field as the default constraint (typically `^`-qualified).

### 4.3 Generics and Monomorphization

Salt uses monomorphization for generics (like Rust and C++). This creates a subtle challenge for the resolver: **a generic function's Z3 contracts cannot be fully verified until the concrete type arguments are known**. The resolver handles this in two phases:

1. **Shape resolution** — The solver resolves all concrete types and trait bounds. Generic dependencies are resolved to their **monomorphized** instances. If package `A` uses `Vec<u64>` from package `B`, and package `C` uses `Vec<String>` from `B`, the resolver treats these as separate concrete compilations (both are cached — see Section 6).

2. **Deferred verification** — Generic functions from dependencies carry their `requires`/`ensures` clauses as AST fragments attached to the monomorphized instance. The Z3 verifier checks these at the call site once concrete types are substituted. This is described in detail in Section 5.

### 4.4 Resolution Algorithm (Pseudocode)

```
function resolve(root_package, constraints):
    partial = empty PartialSolution
    queue = [root_package]
    
    while queue is not empty:
        pkg = queue.pop()
        
        if pkg is already decided in partial:
            continue
        
        // Collect all constraints on pkg from dependents
        terms = collect_incompatibilities(partial, pkg)
        
        // Ask PubGrub: which version(s) satisfy all terms?
        version = select_best_version(pkg, terms)
        
        if version is None:
            // Conflict — derive a new incompatibility and backjump
            cause = conflict_analysis(partial, pkg)
            partial.add_incompatibility(cause)
            backjump(partial, cause)
        else:
            partial.decide(pkg, version)
            
            // Queue transitive dependencies
            for dep, dep_constraint in get_deps(pkg, version):
                enqueue(dep)
                partial.add_term(dep, dep_constraint)
    
    return partial.decisions
```

### 4.5 Workspace Resolution

A workspace is a set of sibling packages under a common root that share a single lockfile. Workspaces are defined implicitly: if `salt.toml` contains a `[workspace]` section listing member directories, all members resolve against the same dependency set to prevent diamond conflicts.

```toml
[workspace]
members = ["core/", "cli/", "daemon/"]
```

Workspaces are a post-v0.1 feature but the manifest schema should reserve the key.

## 5. Contract Verification Across Packages

Salt's Z3 verification is what distinguishes it from every prior-art package manager. When package A depends on package B and calls `B::compute(x)`, the compiler must prove that `x` satisfies B's `requires` clauses at the call site in A.

### 5.1 The Verification Boundary

Every package boundary is a **verification checkpoint**:

```salt
// Package "math-lib" (dependency)
pub fn checked_div(a: i32, b: i32) -> i32
    requires(b != 0)          // <-- contract boundary
    ensures(result * b == a)  // <-- contract boundary
{
    return a / b;
}
```

```salt
// Package "my-app" (consumer)
use math_lib::checked_div;

fn safe_half(x: i32) -> i32 {
    // This call is OK — Z3 can prove x != 0? No!
    // sp add would catch this.
    return checked_div(100, x);  // ❌ requires(b != 0) unproven at call site
}

fn half_of_42() -> i32 {
    return checked_div(42, 2);  // ✅ requires(2 != 0) trivially true
}
```

### 5.2 How Cross-Package Verification Works

Dependencies are compiled in two modes:

**Mode 1: Interface compilation** (happens during `sp add` / `sp lock`)
- The dependency's source is parsed, type-checked, and its Z3 contracts are extracted.
- The monomorphized function signatures and their Z3 `requires`/`ensures` clauses are stored in a **contract summary file** (`.salt-contracts/`).
- No machine code is generated yet.

**Mode 2: Full compilation** (happens during `sp build`)
- The consumer is compiled with the dependency's contract summaries loaded into the Z3 context.
- For each call to a dependency function, the compiler emits a Z3 proof obligation: `requires(args) AND ensures_satisfied_by_implementation`.
- If Z3 cannot prove the `requires` clause, compilation **fails** with a trace of the unprovable condition.

### 5.3 Contract Summary Format

Contract summaries are stored as SIR (Salt Intermediate Representation) fragments in `.salt-contracts/`. A summary for a single function looks like:

```
// .salt-contracts/math-lib/v0.2.0/checked_div.jsonc
{
  "package": "math-lib",
  "version": "0.2.0",
  "commit": "e7f3a9b1c2d5e8f4a0b3c6d7e9f1a2b3c4d5e6f7",
  "exports": {
    "checked_div": {
      "kind": "function",
      "generic_params": [],
      "args": [
        { "name": "a", "type": "i32" },
        { "name": "b", "type": "i32" }
      ],
      "return_type": "i32",
      "requires": [
        { "condition": "b != 0", "z3_encoding": "(not (= b 0))" }
      ],
      "ensures": [
        { "condition": "result * b == a", "z3_encoding": "(= (* result b) a)" }
      ]
    }
  }
}
```

Generic functions store their contracts with symbolic type variables. At monomorphization time, the variables are replaced with concrete types before Z3 verification:

```json
{
  "first": {
    "kind": "function",
    "generic_params": ["T"],
    "args": [
      { "name": "vec", "type": "Vec<T>" },
      { "name": "default", "type": "T" }
    ],
    "return_type": "T",
    "requires": [
      { "condition": "vec.len() > 0", "z3_encoding": "(> (len vec) 0)" }
    ]
  }
}
```

### 5.4 Verification Cache

Once `checked_div` is verified at a given call site with concrete arguments, the result is cached. If `my_app` calls `checked_div(100, x)` in two different functions, the second verification reuses the cached Z3 proof. The cache key is:

```
HASH(package_name | function_name | monomorphized_args_type | requires_encoding)
```

### 5.5 Future: Capability-Based Safety Guarantees

Post-v0.1, Salt contracts can also express **capability bounds** — a dependency's `requires` clause can enforce that the caller holds a certain authority (e.g., `requires(cap_net)`). This would let `sp` build a capability graph: package A with `cap_net` can call B's networking functions; package C without it cannot. This is aspirational and depends on Salt's capability system stabilizing.

## 6. Build Cache

`sp` uses a content-addressed build cache under `salt-target/`, similar to Cargo's `target/` directory but keyed by the hash of build inputs rather than by package name + profile.

### 6.1 Directory Layout

```
salt-target/
  debug/
    <cache-key-1>/   # Hash of source + deps + compiler version + flags
      netd
        libnetd.a
        netd.contracts.json
    <cache-key-2>/
      my-app
        my-app
    .fingerprint     # Map: (package, profile) -> cache key
  release/
    <cache-key-3>/
      ...
```

### 6.2 Cache Key Computation

The cache key is BLAKE3 of:

1. **All source files** in the package (recursive hash of `src/` and `salt.toml`)
2. **Contract summaries** of every transitive dependency (the `.salt-contracts/` directory)
3. **Compiler binary hash** (`salt-front` SHA256)
4. **CLI flags** passed to `sp build` (e.g., `--release`, `--target keuos`)
5. **Salt edition** from `salt.toml`

### 6.3 Invalidation

A cache entry is invalid (rebuilt) when:

- Any source file in the package changes (the source tree hash differs)
- A dependency's contract summary changes (new version, new commit)
- The compiler is upgraded (binary hash differs)
- Build flags change

### 6.4 No Shared Cache Across Projects

Unlike Cargo's `target/` sharing via hardlinks or `CARGO_TARGET_DIR`, `sp` v0.1 keeps each project's cache fully local under its own `salt-target/`. A global user-level cache (`~/.cache/salt/`) is deferred to v0.2 after observing real-world usage patterns.

### 6.5 Fingerprint File

`.fingerprint` is a simple JSON map from `(package_name, profile)` to canonical cache key. On `sp build`, `sp` looks up the package's fingerprint; if the computed key matches, it skips compilation and copies the artifact from the cached directory.

```json
{
  "netd:debug": "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0",
  "my-app:debug": "b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1"
}
```

## 7. Registry Protocol

The v0.1 registry is **Git-only**. There is no central server, no database, no API. Publishing a package means pushing a signed semver tag to a Git repository. Discovering a package means cloning or fetching that repository.

### 7.1 Package Repository Layout

A package repository (e.g., `github.com/example/netd`) mirrors the source tree:

```
netd/
  salt.toml
  src/
    lib.salt
    tcp.salt
    ...
  tests/
    integration.salt
  .salt-contracts/
    # Auto-generated contract summaries for dependents
    v0.2.0/
      checked_div.contracts.json
      ...
```

### 7.2 Version Discovery (Tag Convention)

Versions are discovered by listing Git tags matching `vMAJOR.MINOR.PATCH` (e.g., `v0.2.0`, `v1.0.0`). The `sp add` command:

1. Clones or fetches the repository (shallow: `--depth 50` unless a specific rev is needed)
2. Lists all tags matching the `v\d+.\d+.\d+` pattern
3. Sorts by semver precedence
4. Picks the version that best satisfies the declared constraint

```bash
# What "sp add https://github.com/example/netd" does internally:
git clone --depth 50 https://github.com/example/netd .sp-cache/netd
cd .sp-cache/netd
git tag --list 'v*' | grep -E '^v[0-9]+\.[0-9]+\.[0-9]+$'
# Picks v0.2.0 if it's the latest semver-compatible with ^0.2.0
git checkout v0.2.0
# Reads salt.toml, extracts contracts, builds.
```

### 7.3 Tag Signing and Verification

For v0.1.0, tag verification is optional. If a tag is signed with a GPG or SSH key, `sp` can verify the signature against `salt.toml`'s `trusted-keys` section (future). In v0.1, `sp` trusts whatever commit the tag points to.

### 7.4 Source Integrity

`sp` records the **commit SHA** in `salt.lock`, not the tag name. This protects against tag deletion + re-creation (a form of dependency confusion). If a tag is force-pushed, the lockfile's SHA prevents silent upgrade.

### 7.5 Shallow Clone Strategy

`sp` uses shallow clones aggressively:

- `git clone --depth 50 <url>` for version discovery
- `git fetch --depth 1 origin <commit>` if the lockfile specifies a commit not in the shallow history
- The local cache lives in `~/.cache/sp/git/` keyed by the repository URL's canonical form

This means a single `sp build` on a fresh machine fetches at most `N * depth` Git objects where N is the number of dependencies. For typical projects (5-15 dependencies), this is under 1,000 objects.

### 7.6 Path Dependencies

Path dependencies (`{ path = "../my-lib" }`) bypass Git entirely. They are resolved relative to the consuming package's `salt.toml`. The path is recorded in `salt.lock` as a path entry (not a Git commit). Path dependencies are always rebuilt from source — no cache hit from other projects.

### 7.7 Future: Index Registry

For v0.2+, `sp` may support an **index registry** (like crates.io or Go's proxy.golang.org): a Git repository containing a directory of package metadata (name, description, versions, and their commit SHAs). This enables:

- `sp search` for package discovery
- Dependency resolution without cloning every repository
- Verified checksums (`go.sum`-style) for supply-chain security

The v0.1 Git-only protocol is a superset of the index approach: an index is just a special Git repo whose purpose is to point at other Git repos.

## 8. Prior Art Comparison

### 8.1 Cargo (Rust)

| Feature | Cargo | sp |
|---------|-------|----|
| Manifest | `Cargo.toml` (TOML) | `salt.toml` (TOML) |
| Lockfile | `Cargo.lock` | `salt.lock` |
| Resolver | PubGrub (since 2020) | PubGrub |
| Registry | crates.io + Git dependencies | Git-only (v0.1) |
| Build cache | `target/` (package+profile key) | `salt-target/` (content-addressed) |
| Contracts | No | Z3 requires/ensures at package boundaries |
| Generic deps | Monomorphization, no cross-crate contract check | Monomorphization + deferred Z3 verification |

**Key difference**: Cargo has no concept of formal verification. The `[dependencies]` section is purely about code availability, not about safety guarantees. `sp` adds a verification layer on top of dependency resolution.

### 8.2 Go Modules

| Feature | Go Modules | sp |
|---------|-----------|----|
| Manifest | `go.mod` | `salt.toml` |
| Lockfile | `go.sum` (checksums only) | `salt.lock` (commit SHAs + tree hashes) |
| Resolver | Minimal version selection (MVS) | PubGrub |
| Registry | `proxy.golang.org` + direct VCS | Git-only (v0.1) |
| Version tags | `vMAJOR.MINOR.PATCH` | Same convention |

**Key difference**: Go's MVS is simpler than PubGrub and always picks the oldest compatible version (avoiding "why did my build break?" surprises). However, MVS cannot produce detailed conflict explanations. sp chooses PubGrub for the error quality — when a Salt package has conflicting Z3 contract requirements across the dependency tree, the error message is critical.

### 8.3 npm

| Feature | npm | sp |
|---------|-----|----|
| Manifest | `package.json` (JSON) | `salt.toml` (TOML) |
| Lockfile | `package-lock.json` | `salt.lock` |
| Resolver | Greedy (pre-v5), backtracking (v5+) | PubGrub |
| Registry | npmjs.com (central server) | Git-only (v0.1) |
| Nested deps | Flat by default, nested on conflict | Flat (no diamond duplication) |

**Key difference**: npm allows multiple versions of the same package in the dependency tree (nested node_modules). Salt's monomorphization makes this unnecessary — different consumers of a generic dependency share the monomorphized instance. `sp` always flattens to one version per package.

## 9. Appendix: Implementation Phases

### Phase 1: Core (v0.1.0-alpha)

- `salt.toml` parsing and validation
- Git dependency fetching with shallow clone
- `salt.lock` generation and reading
- PubGrub resolver for non-generic dependencies
- Basic `sp build` that calls `salt-front` with resolved source paths
- `sp clean`

### Phase 2: Verification (v0.1.0-beta)

- Contract summary extraction (`.salt-contracts/` generation during `sp add`)
- Cross-package Z3 verification in the compiler
- Verification cache
- `sp add` contract pre-flight check

### Phase 3: Test Runner (v0.1.0-rc)

- `#[test]` annotation support
- `sp test` — compile and run test binaries
- Dev-dependency resolution

### Phase 4: Polish (v0.1.0)

- `sp new` scaffolding
- Content-addressed build cache with `.fingerprint`
- `sp lock` and `--check` for CI
- Error message improvements for resolution failures

### Phase 5: Post-v0.1 (v0.2+)

- Global user-level cache (`~/.cache/sp/`)
- `sp publish` — push tag and create GitHub release
- Index registry for `sp search`
- Workspace support
- Capability-based safety verification
